## 摘要
大语言模型（LLM）编程智能体越来越多地在仓库级别运行，这推动了在现实约束下评估其优化整个代码库能力的基准测试的发展。现有的代码基准测试主要依赖于合成任务、二元正确性信号或单目标评估，限制了其评估整体优化行为的能力。我们提出了 FormulaCode，这是一个用于在大型、真实世界代码库上评估智能体优化的基准测试，具备细粒度、多目标的性能指标。FormulaCode 包含从 GitHub 上的科学计算 Python 仓库中挖掘出的 957 个性能瓶颈，每个瓶颈都配有专家编写的补丁，且平均每个任务包含 264.6 个社区维护的性能工作负载，从而能够评估 LLM 智能体在现实正确性和性能约束下优化代码库的整体能力。我们的评估显示，仓库级、多目标优化对于前沿 LLM 智能体而言仍然是一个重大挑战。项目网站：https://formula-code.github.io

## Abstract
Large language model (LLM) coding agents increasingly operate at the repository level, motivating benchmarks that evaluate their ability to optimize entire codebases under realistic constraints. Existing code benchmarks largely rely on synthetic tasks, binary correctness signals, or single-objective evaluation, limiting their ability to assess holistic optimization behavior. We introduce FormulaCode, a benchmark for evaluating agentic optimization on large, real-world codebases with fine-grained, multi-objective performance metrics. FormulaCode comprises 957 performance bottlenecks mined from scientific Python repositories on GitHub, each paired with expert-authored patches and, on average, 264.6 community-maintained performance workloads per task, enabling the holistic ability of LLM agents to optimize codebases under realistic correctness and performance constraints. Our evaluations reveal that repository-scale, multi-objective optimization remains a major challenge for frontier LLM agents. Project website at: https://formula-code.github.io

## 1. 论文的核心问题与整体含义

*   **研究背景**：大语言模型（LLM）编程智能体正从孤立的函数级合成转向仓库级的编辑与优化。然而，现有的代码基准测试（如 HumanEval, SWE-Bench）主要依赖合成任务、二元正确性信号（通过/失败）或单一目标评估，无法真实衡量智能体在复杂、相互关联的软件生态系统中进行整体优化的能力。
*   **核心问题**：如何评估 LLM 智能体在真实世界的约束下（如多工作负载权衡、架构级决策、维护函数完整性），对大型代码库进行性能优化的能力？
*   **整体含义**：本研究填补了评估智能体“优化能力”的空白，强调了从“代码正确性”向“代码性能与多目标权衡”转变的必要性，揭示了当前前沿模型在仓库级优化任务中与人类专家存在的巨大差距。

## 2. 论文提出的方法论

*   **核心思想**：提出了 **FORMULA CODE** 基准测试，这是一个基于真实 GitHub 仓库挖掘的、包含多工作负载和多目标性能指标的评估框架。其核心在于利用社区维护的性能基准和专家编写的补丁作为参照，评估智能体的优化效果。
*   **关键技术细节**：
    *   **数据构建流程**：
        1.  **仓库抓取**：筛选包含 Airspeed Velocity (ASV) 性能基准测试基础设施的 Python 仓库。
        2.  **属性过滤**：结合规则过滤和 LLM 分类器，识别旨在提升性能的 Pull Requests (PRs)。
        3.  **环境合成**：利用反思型 LLM 智能体自动生成可复现的 Docker 构建脚本。
        4.  **统计验证**：使用 Mann-Whitney U 检验确保专家补丁带来统计显著的性能提升。
    *   **评估指标**：
        *   **加速比**：智能体修改后的代码相对于基线代码的性能比值，使用几何平均数聚合。
        *   **优势**：智能体加速比与专家加速比之差。若为负值，表示表现不如专家。
        *   **分层优势**：按代码层级（模块、类、函数）聚合优势指标，分析优化范围。
        *   **归一化优势**：考虑方差，衡量优化的稳定性。
        *   **成本加权指标**：结合推理成本（Token 消耗）评估性价比。

## 3. 实验设计

*   **数据集**：
    *   **FORMULA CODE**：包含 957 个任务，来自 70 个开源 Python 仓库（如 Pandas, Scikit-learn, SciPy）。
    *   **FORMULA CODE-V**：为了计算资源限制，从主数据集中采样了 108 个任务用于主要评估。
*   **Benchmark 设置**：
    *   每个任务包含一个基线代码库、问题描述、平均 264.6 个社区维护的性能工作负载以及专家补丁。
    *   评估环境基于 Terminal-Bench 框架扩展，支持性能测量和成本核算。
*   **对比方法**：
    *   **模型**：GPT-5, Claude 4.0 Sonnet, Gemini 2.5 Pro, Qwen 3 Coder。
    *   **智能体框架**：Terminus 2 和 OpenHands。
    *   **基准线**：人类专家编写的补丁。

## 6. 论文的主要结论与发现

*   **整体表现不佳**：所有智能体配置均能实现一定的加速，但整体优势均为负值，表明其表现均不如人类专家。
*   **优化范围局限**：智能体更擅长局部或函数级的优化，而在仓库级或模块级的全局优化上表现较弱。
*   **策略偏好**：智能体在并行化或批处理优化上表现较好，但在涉及底层系统调用（如 C 扩展、向量化操作）的优化上表现挣扎。
*   **流行度影响**：智能体在中等流行度的仓库上表现最好，在尾部（低流行度）和高流行度仓库上表现较差。
*   **成本效益**：尽管前沿模型单次调用成本高，但因推理链更短、成功率更高，整体性价比优于开源权重模型。
*   **权衡能力弱**：与专家相比，智能体在多工作负载优化中更容易导致性能回归，难以做出有利的性能-成本权衡。
*   **数据泄露影响有限**：未发现明显的基于训练数据截止日期的性能突变，表明性能差距主要源于能力而非数据泄露。

## 7. 优点

*   **真实性与实用性**：基准数据完全来自真实的 GitHub 仓库和社区维护的工作负载，避免了合成任务的局限性，更贴近实际软件工程场景。
*   **多目标评估体系**：引入了细粒度的性能指标（如分层优势、归一化优势），超越了简单的二元正确性判断，能够深入分析智能体的优化行为。
*   **自动化构建流程**：提出了一套高度自动化的数据集构建管线，支持持续更新，有助于防止基准测试饱和。
*   **深入的分析维度**：不仅给出了排行榜，还深入分析了优化类型、仓库流行度、成本效益等维度，为未来智能体的改进提供了明确方向。

## 8. 不足与局限

*   **语言与领域限制**：目前仅限于 Python 语言和科学计算领域的仓库，尚未覆盖其他编程语言或应用场景。
*   **评估样本量**：受限于计算资源，主要实验仅在 108 个任务的子集上进行，可能无法完全代表完整数据集的复杂性。
*   **智能体架构限制**：仅评估了 Terminus 2 和 OpenHands 两种框架，未涉及树搜索或进化算法等更复杂的智能体架构。
*   **基础设施依赖**：数据集构建依赖于 ASV (Airspeed Velocity) 基准测试工具，这限制了数据源的覆盖范围（仅包含使用该工具的仓库）。
*   **正确性违规率高**：实验显示智能体生成的补丁有超过 50% 因违反正确性约束（单元测试或快照测试）而被回滚，这表明在优化过程中维持功能正确性仍是巨大挑战。
