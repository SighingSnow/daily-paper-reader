## 摘要
随着 AI 编程智能体成为源代码的主要生产者与消费者，软件行业正面临机构知识加速流失的问题。每一次提交虽然捕获了代码差异，却丢弃了其背后的推理过程——即塑造该决策的约束条件、被否决的备选方案以及前瞻性语境。我将这种被丢弃的推理过程称为“决策阴影”。本文提出了 Lore，这是一种轻量级协议，它利用原生 Git trailers 将提交信息重构为自包含的决策记录，承载约束条件、被否决的备选方案、智能体指令及验证元数据。Lore 除 Git 外无需任何基础设施，可通过独立的 CLI 工具进行查询，且可被任何能够运行 Shell 命令的智能体发现。本文对该协议进行了形式化，将其与五种竞争方法进行了对比，针对最强烈的质疑进行了压力测试，并概述了实证验证路径。

## Abstract
As AI coding agents become both primary producers and consumers of source code, the software industry faces an accelerating loss of institutional knowledge. Each commit captures a code diff but discards the reasoning behind it - the constraints, rejected alternatives, and forward-looking context that shaped the decision. I term this discarded reasoning the Decision Shadow. This paper proposes Lore, a lightweight protocol that restructures commit messages - using native git trailers - into self-contained decision records carrying constraints, rejected alternatives, agent directives, and verification metadata. Lore requires no infrastructure beyond git, is queryable via a standalone CLI tool, and is discoverable by any agent capable of running shell commands. The paper formalizes the protocol, compares it against five competing approaches, stress-tests it against its strongest objections, and outlines an empirical validation path.

## 2. 论文提出的方法论

*   **核心思想：** 提出 **Lore 协议**，将 Git 提交信息重构为“Lore 原子”。每个原子不仅是代码变更的标签，而是包含决策推理过程的最小知识单元，永久绑定于对应的代码变更。
*   **关键技术细节：**
    *   **Git Trailers：** 利用 Git 原生的 trailers 机制（键值对格式）在提交信息底部添加结构化元数据。这无需修改 Git 客户端，且所有标准 Git 工具均可读取。
    *   **词汇表：** 定义了一套标准化的 Trailer 键，包括：
        *   `Constraint:`（约束条件）
        *   `Rejected:`（被否决的备选方案及原因）
        *   `Confidence:`（置信度）
        *   `Directive:`（给未来修改者的指令）
        *   `Tested/Not-tested:`（测试覆盖情况）
    *   **CLI 工具：** 提供一个独立的命令行工具 `lore`，支持智能体进行上下文查询（如 `lore context <path>`）、约束收集、反模式过滤和知识序列化。
*   **工作流程：**
    1.  **发现：** 智能体通过 `.lore` 配置或查看日志发现协议。
    2.  **消费：** 修改代码前，通过 CLI 获取历史决策上下文（约束、被拒方案等）。
    3.  **生产：** 智能体在提交代码时，将决策过程序列化为 Lore 格式，存入提交信息。

## 6. 论文的主要结论与发现

*   **结论：** Git 提交信息是一个未被充分挖掘的知识通道。通过 Lore 协议，可以零成本地将决策上下文嵌入到代码变更历史中。
*   **发现：**
    *   现有的 AI 编程工具生成的提交信息通常是“有损压缩”，仅描述了发生了什么，未描述为什么。
    *   Lore 协议具有“优雅降级”的特性：即使没有 CLI 工具，开发者仍可通过原生 Git 命令读取信息；如果不使用协议，项目状态也不会比现在更糟。
    *   该协议解决了知识同步问题，因为知识直接绑定在不可变的提交对象上。

## 7. 优点

*   **零基础设施成本：** 完全基于 Git 原生特性，无需数据库、服务器或复杂的索引系统。
*   **原子性与不可变性：** 知识与代码变更永久绑定，解决了文档与代码不同步的顽疾。
*   **智能体友好：** 设计了专门面向 AI 智能体的消费模型（CLI 查询），利用智能体现有的 Shell 执行能力，无需复杂的 API 集成。
*   **信息密度高：** 相比保存完整的对话记录，Lore 提取的是经过提炼的结构化知识，更适合上下文窗口受限的 LLM。

## 8. 不足与局限

*   **缺乏实证验证：** 论文目前停留在协议设计阶段，未提供实际应用中的效果数据（如是否真能减少 AI 的试错次数）。
*   **依赖智能体质量：** 协议的有效性高度依赖于 AI 智能体（或人类开发者）能否准确、高质量地填写 Trailer 信息。如果智能体只是机械地填充模板，可能会产生“低质量的 Lore 原子”。
*   **粒度限制：** 论文承认 Lore 适合实现层面的决策，对于跨模块、高层的架构决策，ADRs 等独立文档可能仍是更好的选择。
*   **上下文窗口压力：** 虽然进行了结构化压缩，但在历史悠久的大型项目中，海量的 Lore 信息仍可能对智能体的上下文窗口构成挑战。
