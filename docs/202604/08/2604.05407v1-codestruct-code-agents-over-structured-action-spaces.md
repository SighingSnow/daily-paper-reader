## 摘要
基于LLM的代码智能体将代码库视为非结构化文本，通过脆弱的字符串匹配应用编辑，常因格式漂移或模式歧义而失败。我们提议将代码库重构为结构化动作空间，使智能体对命名的AST实体而非文本片段进行操作。我们的框架CODESTRUCT提供了用于检索完整语法单元的readCode，以及用于对语义程序元素应用经语法验证变换的editCode。在SWE-Bench Verified上对六种LLM进行的评估显示，CODESTRUCT将Pass@1准确率提高了1.2-5.0%，同时大多数模型的token消耗减少了12-38%。在基于文本的接口下频繁无法生成有效补丁的模型受益最大：GPT-5-nano提升了20.8%，其空补丁失败率从46.6%降至7.2%。在CodeAssistBench上，我们观察到一致的准确率提升（+0.8-4.4%），成本降低高达33%。我们的结果表明，结构感知接口为代码智能体提供了更可靠的基础。

## Abstract
LLM-based code agents treat repositories as unstructured text, applying edits through brittle string matching that frequently fails due to formatting drift or ambiguous patterns. We propose reframing the codebase as a structured action space where agents operate on named AST entities rather than text spans. Our framework, CODESTRUCT, provides readCode for retrieving complete syntactic units and editCode for applying syntax-validated transformations to semantic program elements. Evaluated on SWE-Bench Verified across six LLMs, CODESTRUCT improves Pass@1 accuracy by 1.2-5.0% while reducing token consumption by 12-38% for most models. Models that frequently fail to produce valid patches under text-based interfaces benefit most: GPT-5-nano improves by 20.8% as empty-patch failures drop from 46.6% to 7.2%. On CodeAssistBench, we observe consistent accuracy gains (+0.8-4.4%) with cost reductions up to 33%. Our results show that structure-aware interfaces offer a more reliable foundation for code agents.

## 1. 核心问题与整体含义
*   **核心问题**：现有的基于大语言模型（LLM）的代码智能体将代码库视为非结构化的纯文本，通过行号或字符串匹配进行读取和编辑。这种“抽象不匹配”忽略了代码固有的语法和语义结构，导致操作极其脆弱。
*   **具体痛点**：
    *   **读取低效**：智能体要么读取整个文件引入大量无关上下文，干扰推理；要么按行读取导致代码片段被截断。

*   **整体含义**：论文主张将代码库重构为结构化的动作空间，让智能体直接操作抽象语法树（AST）实体，而非文本片段，从而从根本上解决交互脆弱性和低效问题。

## 2. 方法论：核心思想与关键技术
*   **核心思想**：提出 **CODESTRUCT** 框架，将代码库解析为AST，暴露命名的程序实体（如文件、类、函数）作为可寻址的节点。智能体通过两个核心原语与代码交互，实现语义意图与文本实现的解耦。
*   **关键技术细节**：
    *   **readCode（结构化读取）**：
        *   支持目录浏览、文件摘要（返回签名）和实体检索。
        *   通过选择器精准定位AST节点，返回完整的语法单元，避免了无关代码干扰和代码截断问题。
    *   **editCode（结构化编辑）**：
        *   支持插入、替换、删除三种操作。
        *   通过选择器定位目标节点，自动处理缩进，并在提交更改前进行语法验证。
        *   若编辑会导致语法错误则拒绝执行，保证了代码库的完整性。
*   **算法流程**：
    *   **读取流程**：输入路径和选择器 -> 解析AST -> 模糊匹配实体 -> 返回完整实现代码。
    *   **编辑流程**：输入路径、操作类型、选择器和新代码 -> 解析AST -> 定位节点 -> 应用变换 -> 语法检查 -> 写入文件。

## 7. 优点
*   **创新性强**：首次将AST结构作为LLM智能体的主要交互接口，而非辅助信息，从根本上解决了文本交互的脆弱性。
*   **鲁棒性高**：editCode的语法验证机制保证了代码库不会因编辑而崩溃，解决了文本编辑的痛点。
*   **效率显著**：通过精准定位和避免冗余重写，大幅降低了Token消耗和推理成本。
*   **易于集成**：基于MCP协议实现，无需修改智能体的规划逻辑即可无缝集成。
