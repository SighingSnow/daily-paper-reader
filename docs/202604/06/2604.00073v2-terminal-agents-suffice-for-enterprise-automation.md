## 摘要
人们对构建能够与数字平台交互、从而自主执行有意义企业任务的智能体的兴趣日益浓厚。探索的方法包括基于模型上下文协议 (MCP) 等抽象构建的工具增强型智能体，以及通过图形界面操作的 Web 智能体。然而，鉴于其成本和运营开销，目前尚不清楚此类复杂的智能体系统是否确有必要。我们认为，仅配备终端和文件系统的编码智能体可以通过直接与平台 API 交互，更有效地解决许多企业任务。我们在多种现实系统中评估了这一假设，结果表明这些底层终端智能体的表现与更复杂的智能体架构相当甚至更优。我们的研究结果表明，简单的编程接口结合强大的基础模型，足以实现实用的企业自动化。

## Abstract
There has been growing interest in building agents that can interact with digital platforms to execute meaningful enterprise tasks autonomously. Among the approaches explored are tool-augmented agents built on abstractions such as Model Context Protocol (MCP) and web agents that operate through graphical interfaces. Yet, it remains unclear whether such complex agentic systems are necessary given their cost and operational overhead. We argue that a coding agent equipped only with a terminal and a filesystem can solve many enterprise tasks more effectively by interacting directly with platform APIs. We evaluate this hypothesis across diverse real-world systems and show that these low-level terminal agents match or outperform more complex agent architectures. Our findings suggest that simple programmatic interfaces, combined with strong foundation models, are sufficient for practical enterprise automation.

## 1. 论文的核心问题与整体含义

*   **研究背景**：随着大语言模型（LLM）的发展，企业自动化正从简单的文本生成转向能够感知系统状态、推理业务上下文并执行操作的智能体。目前主流架构主要分为两类：一是基于图形用户界面（GUI）的 Web 智能体，二是基于工具增强（如 Model Context Protocol, MCP）的智能体。
*   **核心问题**：现有的复杂智能体架构引入了额外的抽象层，带来了高昂的成本和运维开销。GUI 智能体对界面变化敏感且推理链长，工具增强型智能体则受限于预定义工具的表达能力。论文质疑这种复杂性是否必要，探讨在已有稳定 API 的情况下，是否可以通过更简单的编程接口实现高效的企业自动化。
*   **整体含义**：论文主张“大道至简”，提出仅配备终端和文件系统的编码智能体，通过直接调用平台 API，即可在无需复杂抽象层的情况下，更高效、更低成本地解决企业自动化任务。

## 2. 论文提出的方法论

*   **核心思想**：提出 **StarShell**，一种极简的终端智能体。它不依赖预定义的工具注册表或浏览器自动化，而是像人类开发者一样，通过命令行直接与系统交互。
*   **关键技术细节**：
    *   **交互接口**：智能体仅拥有两个核心接口——终端（用于执行 Shell 命令、调用 `curl` 等）和文件系统（用于存储文档、缓存中间结果、持久化技能）。
    *   **执行循环**：采用 REPL（Read-Eval-Print Loop）风格的循环，即“推理 -> 生成代码/命令 -> 执行 -> 观察输出 -> 修正”。
    *   **动态发现与适应**：不同于 MCP 智能体受限于预定义 Schema，StarShell 可以动态读取 API 文档、检查响应结构，灵活构造请求载荷，甚至编写脚本处理复杂逻辑。
    *   **技能持久化**：智能体可以将解决问题的流程、API 调用模式或避坑指南写入本地文件系统，形成可复用的“技能库”，在后续任务中直接调用，降低探索成本。

## 6. 论文的主要结论与发现

*   **性能优越**：终端智能体在大多数场景下成功率匹配或超越 Web 智能体，且显著优于 MCP 智能体。
*   **成本优势**：终端智能体的推理成本远低于 Web 智能体（通常低 5 倍以上），因为其避免了处理庞大 DOM 树和截图的开销。
*   **MCP 的局限性**：MCP 智能体受限于工具定义的覆盖范围和参数灵活性，即便在可行任务子集上，其表现仍不如终端智能体，因为工具接口往往比底层 API 更僵化。
*   **文档的双刃剑效应**：外部文档并不总是有益。对于结构清晰、任务导向的文档（如 ERPNext），文档有助于发现非显式字段；但对于参考型文档（如 ServiceNow），检索过程可能浪费 Token 甚至误导策略。
*   **技能积累的价值**：允许智能体将经验持久化到文件系统，能显著降低后续任务的成本（最高降低 43.7%）并提升成功率，尤其对于 API 风格不熟悉的平台效果明显。

## 7. 优点

*   **回归本质**：摒弃了繁琐的中间层，利用 API 的原生灵活性和编程能力，证明了“简单接口 + 强模型”的高效性。
*   **鲁棒性强**：终端智能体能够通过试错、编写脚本等方式处理 Shell 引号转义、API 端点错误等底层问题，表现出比 MCP 更强的错误恢复能力。
*   **实验设计严谨**：不仅对比了最终指标，还深入分析了失败原因（如 API 错误分布）、文档影响的机制、技能积累的模式等，提供了丰富的定性洞察。
*   **实用价值高**：该方案部署门槛低，无需维护复杂的浏览器环境或工具封装，仅需 API 访问权限即可运行，适合实际企业环境落地。

## 8. 不足与局限

*   **交互盲区**：终端智能体无法处理纯 UI 交互的任务，例如拖拽式工作流编辑、读取渲染后的图表数据、或需要浏览器 Session Cookie 的操作（如模拟用户登录）。
*   **文档依赖性**：效果受限于平台文档的质量。若文档缺乏结构化或未包含关键 API 细节，智能体可能陷入低效探索。
*   **安全性考量**：终端智能体拥有更广泛的执行权限（如直接读写文件、执行任意 API 调用），对权限控制和审计机制提出了更高要求。
*   **任务范围**：目前的 Benchmark 主要集中在单平台操作，尚未充分覆盖跨平台协调、长周期任务规划等更复杂的企业级场景。
