## 摘要
从高级语言生成高性能可执行文件对于广泛领域的软件性能至关重要。现代编译器通过在逐渐降低的抽象层次上应用一系列经过充分研究的优化来执行此任务，但可能会错失那些需要对程序目的进行高级推理的优化机会。近期研究提出使用大语言模型（LLM）来填补这一空白。虽然LLM在某些程序上可以实现显著的加速，但它们经常生成不正确的代码。在本文中，我们提出了一种方法来平衡传统编译器优化的正确性与基于LLM代码生成的“创造性”：编译器-LLM协作。我们的方法在多个抽象层次上将现有的编译器优化遍与基于LLM的代码生成相结合，保留了这两类代码优化的最佳特性。我们通过一个多智能体系统实现了该方法，该系统包括：(1) 针对每个抽象层次的基于LLM的优化智能体，(2) 作为工具的单个编译器组件，(3) 一个用于探测生成代码正确性和性能的基于LLM的测试生成智能体，以及 (4) 一个编排其他组件的引导LLM。该策略支持在多个抽象层次上对输入程序进行基于LLM的优化，并引入了一种在各层次之间分配计算预算的方法。我们广泛的评估表明，编译器-LLM协作优于现有的编译器优化和特定层次的基于LLM的基线，实现了高达1.25倍的加速。

## Abstract
Generating performant executables from high level languages is critical to software performance across a wide range of domains. Modern compilers perform this task by passing code through a series of well-studied optimizations at progressively lower levels of abstraction, but may miss optimization opportunities that require high-level reasoning about a program's purpose. Recent work has proposed using LLMs to fill this gap. While LLMs can achieve large speedups on some programs, they frequently generate code that is incorrect. In this work, we propose a method to balance the correctness of conventional compiler optimizations with the ``creativity'' of LLM-based code generation: compiler-LLM cooperation. Our approach integrates existing compiler optimization passes with LLM-based code generation at multiple levels of abstraction, retaining the best features of both types of code optimization. We realize our approach with a multi-agent system that includes (1) LLM-based optimization agents for each level of abstraction, (2) individual compiler constituents as tools, (3) an LLM-based test generation agent that probes the correctness and performance of generated code, and (4) a guiding LLM that orchestrates the other components. The strategy enables LLM-based optimization of input programs at multiple levels of abstraction and introduces a method for distributing computational budget between levels. Our extensive evaluation shows that compiler-LLM cooperation outperforms both existing compiler optimizations and level-specific LLM-based baselines, producing speedups up to 1.25x.

## 1. 论文的核心问题与整体含义
*   **研究背景**：现代编译器（如 LLVM）在生成高效可执行代码方面表现出色，但其优化主要依赖于预定义的、确定性的低层次规则（如死代码消除、向量化），往往错失需要理解程序高层意图的优化机会。
*   **现有挑战**：大语言模型（LLM）具备高层推理和代码生成的“创造性”，能够发现编译器遗漏的优化点，但 LLM 经常生成语法错误或逻辑错误的代码，正确性难以保证。
*   **核心问题**：如何平衡传统编译器优化的“正确性”与 LLM 代码生成的“创造性”，以实现超越现有编译器性能的高效代码生成？
*   **整体含义**：本文提出了一种编译器与 LLM 协作的框架，旨在通过多智能体系统，在保证代码正确性的前提下，挖掘跨抽象层级的深度优化潜力。

## 2. 论文提出的方法论
*   **核心思想**：构建一个名为 **ACCLAIM** 的多智能体系统，将编译器的各个阶段（前端、中端、后端）作为工具，与不同抽象层级的 LLM 优化智能体交错执行，通过“引导智能体”动态决策优化路径。
*   **关键定义**：
    *   **重写**：在同一抽象层级内对代码进行变换（如 C 源码到 C 源码）。
    *   **降低**：将代码从高抽象层级转换为低抽象层级（如 C 源码到 LLVM IR）。
    *   **编译器**：由一系列重写和降低操作组成的序列。
*   **系统架构**：
    1.  **引导智能体**：系统的核心调度者。它根据当前上下文和计算预算，决定调用编译器组件还是 LLM 优化智能体，并处理反馈循环。
    2.  **特定层级优化智能体**：针对三个层级（C 源码、LLVM IR、x86 汇编）分别设计的 LLM 智能体，负责在各自层级内进行代码重写优化。
    3.  **测试智能体**：生成测试脚本，验证优化后代码的正确性并测量性能，提供反馈信号。
    4.  **编译器组件**：作为工具被调用，执行标准的编译流程。
*   **算法流程**：
    *   输入源代码后，引导智能体根据预算选择操作（如“调用源码优化智能体”或“调用编译器前端”）。
    *   若调用优化智能体，该智能体执行并行采样（生成 $n$ 个变体）和迭代反馈（$k$ 次循环），选出最佳代码。
    *   测试智能体验证代码正确性与性能，结果反馈给引导智能体。
    *   引导智能体根据反馈决定下一步操作（如回溯、重试或进入下一层级），直到预算耗尽或达到最优。

*   **基准**：以 **Clang -O3** 编译出的代码性能作为基准。
*   **对比方法**：
    1.  **Source-only Agent**：仅在源码层级进行 LLM 优化。
    2.  **IR-only Agent**：仅在 LLVM IR 层级进行 LLM 优化。
    3.  **Assembly-only Agent**：仅在汇编层级进行 LLM 优化。
    4.  **Single-level portfolio**：上述三种单层级方法的简单组合。
*   **评估指标**：相对于基准的加速比。

## 6. 论文的主要结论与发现
*   **性能提升**：ACCLAIM 实现了平均 **1.25 倍** 的加速，优于所有单层级基线方法。在个别案例中实现了高达 1384 倍的加速。
*   **层级贡献**：源码级优化贡献最大，但多层级协作能发现单层级无法捕捉的优化机会（如 IR 层级的指令替换结合编译器的向量化）。
*   **策略有效性**：迭代反馈比单纯的并行采样更有效；增加计算预算能进一步提升性能。
*   **模型依赖**：Claude 3.7 Sonnet 显著优于开源模型，主要归因于其更强的工具调用能力和代码生成正确率。

## 7. 优点
*   **创新性架构**：首次提出了在编译管道的多个抽象层级间动态交错执行 LLM 优化与编译器优化的框架，打破了传统编译器单向流动的限制。
*   **无需训练**：该方法不需要针对特定任务进行模型微调，直接利用现有 LLM 和编译器工具，具有较好的通用性和即插即用性。
*   **自动化闭环**：集成了测试生成智能体，实现了从优化到验证的自动化闭环，有效缓解了 LLM 生成错误代码的问题。

## 8. 不足与局限
*   **计算成本高昂**：每次优化需要多次 LLM 推理调用（$b \times n \times k$），时间和金钱成本较高，可能不适用于一次性运行的程序。
*   **正确性验证局限**：依赖测试用例验证正确性，无法保证 100% 的语义等价性（可能存在测试未覆盖的 Bug）。

*   **数据集偏差**：实验基于 CodeNet 竞赛编程数据集，可能无法完全代表工业界复杂软件系统的优化场景。
