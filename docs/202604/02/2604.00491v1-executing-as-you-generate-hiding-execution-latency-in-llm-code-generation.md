## 摘要
当前的基于LLM的代码智能体遵循串行执行范式：模型首先生成完整代码，随后调用解释器执行该代码。这种顺序工作流导致执行器在生成阶段空闲，而生成器在执行阶段空闲，从而造成了不必要的端到端延迟。我们观察到，与人类开发者不同，LLM顺序生成代码token且无需修改，这使得在代码生成的同时执行代码成为可能。我们将这种并行执行范式形式化，将其建模为包含生成、检测和执行的三级流水线，并推导出了表征其加速潜力和运行机制的闭式延迟界限。随后，我们提出了Eager，这是一个具体实现，具有基于AST的分块、带门控执行的动态批处理以及早期错误中断等特性。我们在四个基准测试、七个LLM和三种执行环境中对Eager进行了评估。结果表明，在七个LLM和四个基准测试中，Eager将非重叠执行延迟降低了高达99.9%，将端到端延迟降低了高达55%。

## Abstract
Current LLM-based coding agents follow a serial execution paradigm: the model first generates the complete code, then invokes an interpreter to execute it. This sequential workflow leaves the executor idle during generation and the generator idle during execution, resulting in unnecessary end-to-end latency. We observe that, unlike human developers, LLMs produce code tokens sequentially without revision, making it possible to execute code as it is being generated. We formalize this parallel execution paradigm, modeling it as a three-stage pipeline of generation, detection, and execution, and derive closed-form latency bounds that characterize its speedup potential and operating regimes. We then present Eager, a concrete implementation featuring AST-based chunking, dynamic batching with gated execution, and early error interruption. We evaluate Eager across four benchmarks, seven LLMs, and three execution environments. Results show that Eager reduces the non-overlapped execution latency by up to 99.9% and the end-to-end latency by up to 55% across seven LLMs and four benchmarks.

## 1. 论文的核心问题与整体含义
*   **研究背景**：当前基于大语言模型（LLM）的代码智能体通常采用“串行执行”范式，即模型必须先生成完整代码，随后解释器才能执行。这种模式导致生成阶段执行器闲置，执行阶段生成器闲置，造成了不必要的端到端延迟。
*   **核心问题**：如何消除这种串行等待带来的延迟，提升代码生成与执行的整体效率？
*   **核心洞察**：与人类开发者不同，LLM 是逐个 token 顺序生成代码且不会回溯修改。这意味着代码一旦生成就已确定，无需等待全部完成即可开始执行。
*   **整体含义**：论文提出了一种“并行执行”范式，旨在将代码执行隐藏在代码生成的过程中，从而显著降低用户感知的端到端延迟。

## 2. 论文提出的方法论
*   **核心思想**：将代码生成与执行建模为一个三阶段流水线，实现“边生成边执行”。
    *   **生成**：LLM 作为生产者流式生成 token。
    *   **检测**：实时解析 token 流，识别出可执行的代码块。
    *   **执行**：消费者进程立即执行已识别的代码块。
*   **关键技术细节**：
    *   **基于 AST 的分块**：利用 Python 抽象语法树（AST）解析器，增量识别完整的语句块。针对函数定义等可能存在歧义的情况，采用“前瞻”策略，确保分块的语法完整性和语义独立性。
    *   **带门控执行的动态批处理**：
        *   **动态批处理**：当执行速度慢于生成速度时，将队列中的多个代码块合并执行，分摊每次调用的启动开销。
        *   **门控执行**：跳过执行时间可忽略不计的代码块（如函数声明），将其与后续有实际计算量的代码块合并执行，避免无谓的开销。
    *   **早期错误中断**：一旦某个代码块运行时出错，立即终止 LLM 的生成过程。这不仅节省了生成后续无效代码的时间，还提供了更早的错误反馈。
*   **理论建模**：论文建立了数学模型推导延迟界限。
    *   定义了生成主导和执行主导两种机制。
    *   推导证明了在生成主导（生成时间 > 执行时间）的常见场景下，并行执行能显著降低延迟；即使在执行主导场景，开销也可控。

## 6. 论文的主要结论与发现
*   **延迟降低显著**：Eager 能够隐藏 83%-100% 的执行时间（NEL 降低）。在真实 LLM 实验中，端到端延迟最高降低了 55%（遇到错误时）和 37%（无错误时）。
*   **生成速度的影响**：生成速度越快（TPS 越高），执行时间占比越大，并行带来的 E2EL 收益越明显。
*   **修复能力提升**：早期错误中断机制不仅减少了延迟，还提高了代码修复的成功率（在数据分析类任务上提升 2.1 到 44.3 个百分点）。原因在于避免了模型被错误代码后的无效上下文“锚定”。
*   **特例情况**：在 GitChameleon（版本特定代码生成）任务中，早期中断略微降低了修复成功率，因为被截断的代码可能包含关键的 API 版本上下文信息。

## 7. 优点
*   **创新性范式**：首次形式化了 LLM 代码生成的并行执行范式，打破了传统的串行思维定式。
*   **理论支撑扎实**：通过流水线建模推导了延迟界限，为系统设计提供了理论依据。
*   **工程实现巧妙**：AST 分块、动态批处理和门控机制有效解决了流式执行中的技术难题（如启动开销、状态维护）。
*   **额外收益**：意外发现早期中断对代码修复有正向促进作用，提升了系统的鲁棒性。

## 8. 不足与局限
*   **语言局限性**：目前实现主要针对 Python，虽然原理通用于解释型语言，但尚未在编译型语言（如 C++/Java）上验证。
*   **任务类型限制**：实验主要针对短/中等长度的脚本任务。对于计算密集型（执行时间远大于生成时间）的任务，并行效果有限。
*   **特定场景下的副作用**：在版本特定代码生成任务中，早期中断可能丢失关键上下文，导致修复效果下降。
