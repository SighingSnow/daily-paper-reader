## 摘要
企业用户越来越依赖 AI 智能体通过自然语言查询其数据。然而，构建可靠的数据智能体仍然困难重重，因为现实世界的数据通常碎片化地分布在多个异构数据库系统中，且存在引用不一致以及信息深埋于非结构化文本中的问题。现有的基准测试仅解决了该问题的局部环节——例如，将自然语言问题转化为 SQL 查询，或基于上下文中提供的小型表格回答问题——但并未评估跨多个数据库系统进行数据集成、转换和分析的完整流程。为了填补这一空白，我们提出了数据智能体基准测试（DAB），该基准基于对六个行业的企业数据智能体工作负载的形成性研究。DAB 包含 54 个查询，涵盖 12 个数据集、9 个领域和 4 个数据库管理系统。在 DAB 上，表现最好的前沿模型（Gemini-3-Pro）仅达到了 38% 的 pass@1 准确率。我们对五个前沿大语言模型进行了基准测试，分析了它们的失效模式，并为未来的数据智能体开发提炼了经验启示。我们的基准测试和实验代码已发布于 github.com/ucbepic/DataAgentBench。

## Abstract
Users across enterprises increasingly rely on AI agents to query their data through natural language. However, building reliable data agents remains difficult because real-world data is often fragmented across multiple heterogeneous database systems, with inconsistent references and information buried in unstructured text. Existing benchmarks only tackle individual pieces of this problem -- e.g., translating natural-language questions into SQL queries, answering questions over small tables provided in context -- but do not evaluate the full pipeline of integrating, transforming, and analyzing data across multiple database systems. To fill this gap, we present the Data Agent Benchmark (DAB), grounded in a formative study of enterprise data agent workloads across six industries. DAB comprises 54 queries across 12 datasets, 9 domains, and 4 database management systems. On DAB, the best frontier model (Gemini-3-Pro) achieves only 38% pass@1 accuracy. We benchmark five frontier LLMs, analyze their failure modes, and distill takeaways for future data agent development. Our benchmark and experiment code are published at github.com/ucbepic/DataAgentBench.

## 1. 论文的核心问题与整体含义

*   **研究背景**：企业用户日益依赖 AI 智能体通过自然语言查询数据。然而，现实世界的企业数据通常具有**碎片化**（分散在多个孤岛）、**异构性**（不同数据库系统）以及**非结构化**（信息埋藏在文本中）的特征。
*   **核心问题**：现有的基准测试（如 Text-to-SQL 或 Table QA）仅关注单一环节（如生成 SQL 或基于小表格回答），无法评估跨多数据库系统集成、转换和分析数据的**端到端流程**。这导致缺乏有效手段来衡量和改进数据智能体在真实复杂场景下的能力。
*   **整体含义**：本文旨在填补这一空白，提出首个面向复杂数据任务的基准测试，揭示当前前沿大模型在处理真实企业级数据问题时的局限性，并为未来数据智能体的开发指明方向。

## 2. 论文提出的方法论

*   **核心思想**：构建 **Data Agent Benchmark (DAB)**，一个基于真实企业工作负载特征设计的基准测试，旨在评估智能体在多数据库环境下的端到端数据处理能力。
*   **形成性研究**：通过对 6 个行业的企业客户进行访谈，识别出真实数据查询的四大难点属性：
    1.  **多数据库集成**：需跨不同查询语言（如 SQL 和 MongoDB）的数据库进行操作。
    2.  **格式错误的连接键**：同一实体在不同数据库中标识符不一致（如前缀不同、空格差异），需智能体检测并清洗。
    3.  **非结构化文本转换**：需从自由文本字段中提取结构化数值（如从评论中提取评分）。
    4.  **领域知识**：需依赖数据库模式之外的专业知识（如金融计算公式）。
*   **基准构建流程**：
    1.  **数据收集**：选取 12 个开源数据集，覆盖新闻、电商、金融等 9 个领域。
    2.  **数据扰动**：模拟真实混乱数据。移除直接回答问题的列，将其信息嵌入文本字段；修改连接键格式使其不一致。
    3.  **多库分发**：将每个数据集的表分发到至少 2 个不同的数据库管理系统（PostgreSQL, MongoDB, SQLite, DuckDB）。
    4.  **任务设计**：设计了 54 个自然语言查询，每个查询均涉及多数据库集成及至少一项其他难点属性。
*   **智能体架构**：采用 ReAct 模式，配备四个工具：`list_db`（列出表）、`query_db`（执行 SQL/Mongo 查询）、`execute_python`（执行 Python 代码处理数据）、`return_answer`（返回最终答案）。

## 3. 实验设计

*   **数据集 / 场景**：DAB 基准，包含 12 个数据集、54 个查询、4 种数据库系统。
*   **评估对象**：5 个前沿大语言模型，包括闭源模型 GPT-5.2, GPT-5-mini, Gemini-3-Pro, Gemini-2.5-Flash 和开源模型 Kimi-K2。
*   **对比方法**：
    *   **基线方法**：基于 ReAct 模式的通用智能体。
    *   **对比系统**：PromptQL（Hasura 公司的生产级数据智能体平台），用于评估专用基础设施对性能的提升。
*   **评估指标**：
    *   **Pass@k**：估计 k 次尝试中至少成功一次的概率（主要关注 Pass@1）。
    *   **成本**：API 调用费用。
    *   **轨迹统计**：延迟、迭代次数、工具调用次数。

## 4. 资源与算力

*   **算力资源**：文中未提及模型训练所需的 GPU 资源，因为实验主要评估的是现成的大模型 API 服务。
*   **实验开销**：实验共运行了 13,500 次试验（54 个查询 × 5 个模型 × 50 次试验），总 API 成本约为 **3,150 美元**。
*   **运行限制**：每次试验限制最大 100 次迭代和 1 小时的挂钟时间。

## 5. 实验数量与充分性

*   **实验规模**：
    *   主实验：5 个模型 × 54 个查询 × 50 次重复 = 13,500 次轨迹。
    *   失败模式分析：人工检查 30 条轨迹，并使用 GPT-5 对 1,147 条失败轨迹进行自动分类。
    *   案例研究：对比 PromptQL 与 ReAct 基线（使用 Claude-Opus-4.6 模型，每个查询 5 次试验）。
*   **充分性与客观性**：
    *   实验样本量较大（50 次重复），统计结果具有较高可信度。
    *   评估涵盖了不同厂商、不同规模的模型，具有代表性。
    *   失败模式分析结合了人工定性分析与 LLM 辅助的大规模标注，方法严谨。
    *   **局限性**：验证脚本采用子串匹配判断答案正确性，可能容忍包含正确答案但也包含错误信息的冗余回答（重召回轻精确）。

## 6. 论文的主要结论与发现

*   **整体性能低下**：表现最好的模型 Gemini-3-Pro 的 Pass@1 准确率仅为 **38%**，即使 Pass@50 也仅达到 69%。有一个数据集甚至被所有模型“零封”。
*   **主要失败模式**：85% 的错误源于**规划错误（FM2）**或**实施错误（FM4）**，而选错数据源（FM3）的情况很少。这表明智能体知道“去哪找数据”，但不知道“怎么算”或“代码写不对”。
*   **文本处理短板**：所有模型在处理非结构化文本时均依赖正则表达式，缺乏使用 NLP 工具或 LLM 提取工具的能力，导致在复杂文本解析任务（如专利日期提取）上全军覆没。
*   **成本效益差异**：GPT-5-mini 虽然模型较小，但在成本效益上表现最佳；Kimi-K2 成本高但准确率低。
*   **专用工具的价值**：PromptQL 相比 ReAct 基线提升了 7 个百分点，说明语义层和专用基础设施有助于缓解数据探索的负担，但仍无法解决文本提取难题。

## 7. 优点

*   **真实性与针对性**：基于真实企业工作负载的形成性研究构建基准，捕捉了现有基准忽视的“脏数据”特征（如不一致的键、非结构化文本）。
*   **端到端评估**：突破了单一 SQL 生成的局限，涵盖了从数据发现、集成、清洗到分析的全流程。
*   **深入的错误分析**：不仅给出了准确率数字，还建立了详细的失败模式分类法，深入剖析了智能体的能力瓶颈（规划与实施）。
*   **可复现性**：提供了开源代码和数据集，且基准构建过程（扰动方法）描述清晰。

## 8. 不足与局限

*   **基准规模**：仅包含 54 个查询，相比 Text-to-SQL 等领域的基准（通常包含数千条）规模较小。
*   **验证机制**：答案验证采用子串匹配，可能存在假阳性（即答案包含正确文本但也包含错误内容），未严格验证答案的精确性。
*   **提示词工程**：实验主要使用统一的提示词模板，未针对不同模型的特性进行深度优化，可能低估了部分模型的潜力。
*   **模型覆盖**：受限于 API 额度，未包含 Anthropic Claude 系列模型在主实验中（仅在案例研究中使用了 Claude）。
*   **上下文管理**：实验采用了简单的截断策略来处理长上下文，这可能影响了模型处理大规模数据（如包含 2754 张表的 stockmarket 数据集）的能力。
