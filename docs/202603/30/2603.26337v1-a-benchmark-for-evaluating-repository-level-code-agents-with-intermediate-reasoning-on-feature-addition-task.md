## 摘要
仓库级代码智能体在现实世界的功能添加任务中展现出了巨大的潜力，这使得对其能力的可靠评估变得日益重要。然而，现有的基准主要基于最终测试的正确性将这些智能体作为黑盒进行评估，对于它们如何推理以及故障源于何处提供的见解有限。为了解决这一局限性，我们提出了 RACE-bench，这是一个推理增强型基准，用于评估代码智能体在仓库级功能添加任务上的表现。RACE-bench 包含来自 12 个开源仓库的 528 个真实世界功能添加实例。每个实例都配有可执行补丁验证以及结构化的中间推理真值，涵盖问题理解、文件定位、实现任务和步骤分解。基于这一设计，我们引入了一个双轨评估框架，联合衡量补丁正确性和中间推理质量。我们在 RACE-bench 上评估了三个具有代表性的仓库级代码智能体。在完整基准测试中，不同智能体的解决率在 29% 到 70% 之间。我们的推理层面分析进一步表明，虽然当前的智能体在理解高层意图方面表现良好，但在将意图转化为具体的实现步骤时，其性能显著下降。我们还发现，与成功案例相比，应用成功但测试失败的案例表现出更低的推理召回率（下降 35.7%）和更高的过预测率（增加 94.1%）。这些发现强调了通过考察推理过程的质量，在最终补丁正确性之外评估仓库级代码智能体的重要性。

## Abstract
Repository-level code agents have shown strong promise in real-world feature addition tasks, making reliable evaluation of their capabilities increasingly important. However, existing benchmarks primarily evaluate these agents as black boxes based on final test correctness, providing limited insight into how they reason and where failures arise. To address this limitation, we introduce RACE-bench, a reasoning-augmented benchmark for evaluating code agents on repository-level feature addition tasks. RACE-bench contains 528 real-world feature addition instances from 12 open-source repositories. Each instance is paired with executable patch verification and structured intermediate reasoning ground truth covering issue understanding, file localization, implementation tasks, and step decomposition. Based on this design, we introduce a dual-track evaluation framework that jointly measures patch correctness and intermediate reasoning quality. We evaluate three representative repository-level code agents on RACE-bench. On the full benchmark, Resolved Rates range from 29% to 70% across different agents. Our reasoning-level analysis further shows that while current agents perform well at understanding high-level intent, their performance degrades substantially when translating intent into concrete implementation steps. We also find that apply-success but test-fail cases exhibit lower reasoning recall (35.7% decrease) and higher over-prediction (94.1% increase) compared to successful cases. These findings highlight the importance of evaluating repository-level code agents beyond final patch correctness by examining the quality of their reasoning processes.

*   **核心问题**：现有的评估方式缺乏对智能体中间推理过程的洞察，无法解释智能体为何失败（例如是误解意图、定位错误文件还是计划制定错误），也无法识别那些“碰巧通过测试”但推理错误的案例。

## 2. 论文提出的方法论
*   **核心思想**：构建一个包含结构化“推理真值”的基准，并提出双轨评估框架，联合衡量补丁正确性与中间推理质量。
*   **基准构建**：
    *   **数据来源**：从 12 个流行的开源 Python 仓库中收集合并的 Pull Request (PR)。
    *   **过滤流程**：通过属性过滤（是否有测试、是否关联 Issue）、意图过滤（LLM 分类筛选“新功能”类 PR）、执行过滤（验证测试环境可运行且 FTP/PTP 测试有效），最终筛选出 528 个高质量实例。
    *   **推理真值构建**：将开发流程分解为四个阶段，并利用 LLM 辅助生成与补丁驱动分析相结合的方式构建真值：
        1.  **问题理解**：提取概念解释和目标期望。
        2.  **文件定位**：区分必要代码文件、其他文件和测试文件。
        3.  **问题实现**：识别具体的代码实体（函数/方法）变更及其必要性。
        4.  **步骤分解**：将实现过程抽象为最小化的语义步骤序列（如“引入新功能”、“复用现有语义”等）。
*   **评估框架**：
    *   **补丁验证**：通过执行单元测试（FTP 和 PTP 测试）判断补丁的功能正确性。
    *   **推理评估**：利用 Summarizer LLM 将智能体的交互轨迹标准化，并与推理真值对比。指标包括召回率和过预测率，对于语义判断任务采用 LLM-as-Judge 机制。

## 6. 论文的主要结论与发现
*   **性能差异**：智能体的解决率差异显著（29% - 70%），其中 mini-SWE-Agent 表现最佳。
*   **推理瓶颈**：智能体普遍擅长理解高层意图（Goal Score > 9.2/10），但在将意图转化为具体实现步骤时出现“瀑布式”性能下降，步骤分解是主要瓶颈。
*   **失败模式**：应用成功但测试失败的案例通常伴随着较低的推理召回率（下降 35.7%）和较高的过预测率（增加 94.1%），即修改范围过大或遗漏关键步骤。
*   **智能体风格**：AutoCodeRover 风格保守（低过预测但低召回），TraeAgent 风格激进（高探索但高冗余），mini-SWE-Agent 实现了较好的平衡。
*   **基座模型影响**：基座模型的影响受限于智能体架构。设计良好的架构（如 mini-SWE-Agent）在不同模型下表现更稳定；GPT 模型倾向于生成更精确的修改，减少冗余。

## 7. 优点
*   **评估视角创新**：突破了传统“黑盒”评估的局限，首次引入结构化的中间推理评估，能够诊断智能体失败的具体环节。
*   **数据质量高**：构建流程严谨，包含执行验证和人工审核，确保了测试用例的有效性和推理真值的准确性。
*   **分析深入**：不仅报告了准确率，还深入分析了推理过程中的召回率、过预测率与最终结果的相关性，揭示了“意图-执行”差距。

*   **显性推理依赖**：评估依赖于智能体输出的显性推理轨迹，若智能体内部存在隐性推理未输出，可能无法被完全捕捉。
