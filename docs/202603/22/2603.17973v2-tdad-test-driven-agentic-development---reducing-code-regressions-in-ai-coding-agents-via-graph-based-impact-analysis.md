## 摘要
AI编程智能体能够解决现实世界的软件问题，但它们经常引入回归——即破坏先前通过的测试。当前的基准测试几乎只关注解决率，导致回归行为未被充分研究。本文提出了TDAD（测试驱动的智能体开发），这是一个为AI编程智能体执行变更前影响分析的开源工具。TDAD构建了源代码与测试之间的依赖映射，使得智能体在提交补丁之前知道需要验证哪些测试，并能够进行自我纠正。该映射以轻量级智能体技能的形式提供——即一个智能体在运行时查询的静态文本文件。在SWE-bench Verified上，使用运行于消费级硬件上的两个开放权重模型（Qwen3-Coder 30B，100个实例；Qwen3.5-35B-A3B，25个实例）进行评估，与原始基线相比，TDAD将回归率降低了70%（从6.08%降至1.82%）。相比之下，在没有针对性测试上下文的情况下添加TDD过程指令，反而将回归率增加到了9.94%——比完全不干预还要糟糕。当作为智能体技能部署在不同的模型和框架上时，TDAD将问题解决率从24%提高到32%，证实了呈现上下文信息优于规定过程工作流。所有代码、数据和日志均已在 https://github.com/pepealonso95/TDAD 公开。

## Abstract
AI coding agents can resolve real-world software issues, yet they frequently introduce regressions -- breaking tests that previously passed. Current benchmarks focus almost exclusively on resolution rate, leaving regression behavior under-studied. This paper presents TDAD (Test-Driven Agentic Development), an open-source tool that performs pre-change impact analysis for AI coding agents. TDAD builds a dependency map between source code and tests so that before committing a patch, the agent knows which tests to verify and can self-correct. The map is delivered as a lightweight agent skill -- a static text file the agent queries at runtime. Evaluated on SWE-bench Verified with two open-weight models running on consumer hardware (Qwen3-Coder 30B, 100 instances; Qwen3.5-35B-A3B, 25 instances), TDAD reduced regressions by 70% (6.08% to 1.82%) compared to a vanilla baseline. In contrast, adding TDD procedural instructions without targeted test context increased regressions to 9.94% -- worse than no intervention at all. When deployed as an agent skill with a different model and framework, TDAD improved issue-resolution rate from 24% to 32%, confirming that surfacing contextual information outperforms prescribing procedural workflows. All code, data, and logs are publicly available at https://github.com/pepealonso95/TDAD.

## 2. 论文提出的方法论
*   **核心思想**：提出 TDAD（Test-Driven Agentic Development）工具，通过构建源代码与测试之间的依赖图，在智能体提交补丁前进行“变更前影响分析”。核心洞察是：智能体不需要被告知“如何做 TDD”，而是需要被告知“检查哪些测试”。
*   **关键技术细节**：
    *   **图构建**：解析 Python 仓库构建代码-测试依赖图。节点包括文件、函数、类、测试；边包括包含、调用、导入、测试、继承关系。
    *   **测试链接**：通过命名约定、前缀匹配和目录邻近度算法，将测试文件与源码文件关联。
    *   **影响分析策略**：并行运行四种策略（直接测试、传递调用、文件覆盖、导入关系），计算测试的风险评分。
*   **算法流程**：
    *   评分公式：$score = (1 - c_w) \cdot w_{strategy} + c_w \cdot confidence$。其中 $c_w=0.3$ 为置信度权重，策略权重 $w_{strategy}$ 根据不同策略设定（如直接测试为 0.95，导入为 0.50）。
    *   输出：生成静态的 `test_map.txt` 文件，供智能体通过 `grep` 查询。
*   **智能体集成**：将 TDAD 封装为轻量级技能，包含 `test_map.txt` 和仅 20 行的 `SKILL.md` 指令文件，无需数据库或 API 调用。

## 6. 论文的主要结论与发现
*   **显著降低回归**：TDAD 将测试级回归率降低了 70%（从 6.08% 降至 1.82%），P2P 失败数从 562 降至 155。
*   **TDD 提示词悖论**：仅添加 TDD 过程指令（无图上下文）反而使回归率恶化至 9.94%。原因是冗长的指令挤占了小模型的上下文窗口，且缺乏定位导致盲目修改。
*   **上下文优于流程**：对于小模型，提供“查哪些测试”的上下文信息远比规定“怎么写测试”的流程指令有效。
*   **提升解决率**：在 Phase 2 中，TDAD 作为技能部署，将问题解决率从 24% 提升至 32%，生成率从 40% 提升至 68%。
*   **自我改进能力**：自动改进循环将解决率从 12% 提升至 60%，且回归率保持为 0%。

## 7. 优点

*   **洞察深刻**：揭示了“上下文 > 流程”的设计原则，指出小模型在有限窗口下更需要精准的信息而非冗长的指令。
*   **开源贡献**：提供了完整的代码、数据和日志，工具可通过 pip 安装，复现性强。
